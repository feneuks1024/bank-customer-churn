# Прогнозирование оттока клиентов банка

Проект посвящён анализу и прогнозированию оттока клиентов банка. В работе сравниваются классические модели машинного обучения и алгоритмы градиентного бустинга, исследуется влияние дисбаланса классов и выполняется подбор порога классификации.

## Задача

Построить модель бинарной классификации для определения вероятности ухода клиента из банка:

- `0` --- клиент продолжает пользоваться услугами;
- `1` --- клиент покинул банк.

Такую модель можно использовать для формирования списка клиентов с повышенным риском оттока и дальнейшей работы с ними.

## Данные

Использован набор [Bank Customer Churn](https://www.kaggle.com/datasets/radheshyamkollipara/bank-customer-churn) с платформы Kaggle.

Датасет содержит 10 000 записей о клиентах банка и 18 столбцов, включая:

- кредитный рейтинг;
- страну и пол клиента;
- возраст;
- срок обслуживания;
- баланс;
- количество используемых продуктов;
- наличие банковской карты;
- активность клиента;
- предполагаемую заработную плату;
- целевой признак `Exited`.

Распределение целевого класса:

- около 79,6% клиентов остались;
- около 20,4% клиентов ушли.

Таким образом, в данных присутствует дисбаланс классов.

## Предобработка данных

Из набора были исключены идентификаторы и признаки, которые не должны использоваться для построения основной модели:

- `RowNumber`;
- `CustomerId`;
- `Surname`.

Признаки `Complain`, `Satisfaction Score`, `Card Type` и `Point Earned` также не использовались в итоговом моделировании из-за неясного момента их формирования и возможного риска утечки информации о целевом событии.

Для остальных данных выполнены:

- проверка пропущенных значений;
- One-Hot Encoding категориальных признаков;
- масштабирование числовых признаков для Logistic Regression;
- стратифицированное разделение данных;
- обработка дисбаланса классов;
- подбор индивидуального порога для каждой модели.

## Разделение данных

Данные разделены на три части:

- **train** --- обучение моделей;
- **validation** --- подбор конфигурации и порога классификации;
- **test** --- итоговая независимая оценка качества.

Тестовая выборка не использовалась для выбора моделей и настройки порогов.

## Модели

В проекте сравнивались:

- Logistic Regression;
- Random Forest;
- Gradient Boosting;
- XGBoost;
- LightGBM;
- CatBoost.

## Оценка качества

Использовались следующие метрики:

- Accuracy;
- Precision;
- Recall;
- F1-score;
- ROC-AUC;
- матрица ошибок.

Accuracy не использовалась как единственный критерий качества из-за дисбаланса целевого класса.

## Результаты

Лучший итоговый результат показал **CatBoost** после подбора порога на validation-выборке.

| Модель   |   ROC-AUC | Precision |    Recall |  F1-score | CV ROC-AUC |
|----------|----------:|----------:|----------:|----------:|-----------:|
| CatBoost | **0.878** | **0.710** | **0.571** | **0.633** |  **0.866** |

Полное сравнение всех моделей, их порогов и метрик представлено в ноутбуке.

## Интерпретация результата

- Precision `0.710` означает, что около 71% клиентов, отмеченных моделью как склонные к уходу, действительно относятся к целевому классу.
- Recall `0.571` означает, что модель обнаруживает около 57% всех ушедших клиентов.
- ROC-AUC `0.878` показывает хорошую способность модели разделять клиентов двух классов при различных порогах.

Оптимальный порог следует выбирать с учётом бизнес-стоимости ошибок:

- высокий Recall подходит, когда важно не пропустить клиента с риском ухода;
- высокий Precision подходит, когда предложение по удержанию является дорогим и его необходимо направлять более точной группе клиентов.

## Важность признаков

По результатам анализа Random Forest среди наиболее значимых для прогноза признаков оказались:

- возраст клиента;
- количество используемых продуктов;
- баланс;
- активность клиента.

Важность признаков показывает их вклад в прогноз модели, но не доказывает причинно-следственную связь с уходом клиента.

## Основные выводы

- Данные имеют выраженный дисбаланс классов примерно 80/20.
- Ансамблевые модели показали более высокое качество, чем Logistic Regression.
- CatBoost показал лучший итоговый ROC-AUC.
- Подбор порога позволил настроить баланс между Precision и Recall.
- Для практического использования порог необходимо выбирать с учётом стоимости удержания клиента и потерь от его ухода.

## Запуск проекта

Клонировать репозиторий:

    git clone <ссылка-на-репозиторий>
    cd bank-customer-churn

Создать виртуальное окружение:

    python -m venv .venv

Активировать окружение в Windows:

    .venv\Scripts\activate

Установить зависимости:

    pip install -r requirements.txt

Запустить ноутбук:

    jupyter notebook bank_churn_analysis.ipynb

Файл `Customer-Churn-Records.csv` должен находиться в одной директории с ноутбуком.

## Ограничения

- Проект использует публичный учебный датасет.
- Модель не проверялась на реальных данных конкретного банка.
- Некоторые признаки исключены из-за риска утечки данных и отсутствия точного описания момента их формирования.
- Важность признаков не является доказательством причин оттока.
- Перед внедрением потребуется оценить стоимость ошибок, выбрать бизнес-порог и организовать мониторинг качества модели.
